# hany-rules — 编码约束

**触发方式：自动加载（其他 skill 启动时引用）**

> **核心原则：先理解再动手，有证据才算完，简单即正义。**

以下 11 条规则在所有 hany skill（require / implement / verify / auto / planreview）中全程生效。

## 强制交互规则

> 其他 skill 在执行步骤时，必须按本文件的"执行关联"检查对应规则。违反规则的行为应被拦截并纠正。

## 质量模式

从 `.hany/config.json` 的 `quality` 字段读取，默认 `standard`。

| 模式 | 规则执行力度 |
|------|------------|
| **strict** | 全部 11 条强制检查，违规即中断 |
| **standard**（默认） | 全部 11 条检查，违规需修正后继续 |
| **minimal** | 核心规则（1/2/6/9）强制，其余建议遵守 |

## 概览（必读）

> **[按需加载]** 本文件按规则组织。引用时直接跳读到对应规则编号即可。

### 快速参考卡

| 编号 | 规则 | 核心要求 | 优先级 |
|------|------|---------|--------|
| 1 | 先思考后行动 | 理解后动手 | 高 |
| 2 | 先阅读后编写 | 读完再改 | 高 |
| 3 | 简洁输出但深思熟虑 | 代码简，思考全 | 中 |
| 4 | 优先编辑而非重写 | 改最小范围 | 中 |
| 5 | 避免重复读取 | 不浪费上下文 | 低 |
| 6 | 测试后再宣布完成 | 有证据才算完 | 高 |
| 7 | 避免谄媚和冗余 | 直接专业 | 中 |
| 8 | 保持方案简单直接 | KISS | 中 |
| 9 | 用户指令优先 | 用户 > 规则 | 高 |
| 10 | 从重复错误中学习 | 会话内即刻内化 | 中 |

## 规则 1：先思考后行动 [优先级：高]

理解需求和现有结构后再动手。

| | 示例 |
|---|------|
| **正确** | 收到"加个登录功能"→ 先看项目结构、现有认证方案、技术栈，再动手 |
| **错误** | 收到"加个登录功能"→ 直接创建 `login.ts`，不看已有代码 |

**触发条件**：收到新任务时、需要修改代码时、被要求实现功能时。

**执行关联**：hany-implement Step 0（先想想看）、hany-require Step 0（先想想看）、hany-verify-* Step 0。

## 规则 2：先阅读后编写 [优先级：高]

修改前完整阅读文件，理解上下文。

| | 示例 |
|---|------|
| **正确** | 要改 `utils.ts` → 先 `Read` 整个文件，理解函数关系和导出结构 |
| **错误** | 要改 `utils.ts` → 直接定位到第 50 行 `Edit`，忽略上下文依赖 |

**触发条件**：任何文件修改操作前。

**执行关联**：hany-implement Step 7（并行实现）中 Agent 返回后验证。

## 规则 3：简洁输出但深思熟虑 [优先级：中]

代码简洁清晰，思考全面包括边界情况。

| | 示例 |
|---|------|
| **正确** | `const result = items.filter(Boolean)` + 内部已考虑 null/undefined/空字符串 |
| **错误** | 写 20 行 if-else 处理同一件事；或只处理 happy path 忽略边界 |

**触发条件**：编写新代码、重构、设计方案时。

**执行关联**：hany-implement Step 2（Constitutional 门禁：简单性检查）。

## 规则 4：优先编辑而非重写 [优先级：中]

只改必要部分，保留历史意图。

| | 示例 |
|---|------|
| **正确** | 用 `Edit` 修改第 30-35 行的逻辑，保留其余部分不变 |
| **错误** | `Write` 覆盖整个文件，只为了改一行 |

**触发条件**：修改现有文件时。

**执行关联**：hany-implement Step 3（文件结构映射）中区分修改 vs 新建。

## 规则 5：避免重复读取 [优先级：低]

已读文件不重复读，除非可能被修改。

| | 示例 |
|---|------|
| **正确** | 已读 `config.json`，后续步骤直接引用缓存内容 |
| **错误** | 三个步骤分别 `Read` 同一个未修改的文件 |

**例外**：Agent 并行执行后返回的结果、用户告知已修改的文件、git 操作后的文件。

**执行关联**：所有 skill 的上下文管理。

## 规则 6：测试后再宣布完成 [优先级：高]

验证功能符合预期再结束。

| | 示例 |
|---|------|
| **正确** | "登录功能完成"→ 运行 `npm test`，附带输出 → "验证通过：3 tests passed" |
| **错误** | "登录功能应该没问题" → 没有运行任何命令 |

**铁律**：没有验证命令输出的"完成"声明不算数。

**触发条件**：任何声称"完成"、"已修复"、"通过"的时刻。

**执行关联**：hany-implement Step 8（完成前验证门禁）、hany-verify-small Step 2（功能验证）、hany-verify-project Step 5-6。

## 规则 7：避免谄媚和冗余 [优先级：中]

直接、简洁、专业，不客套。

| | 示例 |
|---|------|
| **正确** | "已修复 null 检查问题。验证：`tsc --noEmit` 通过。" |
| **错误** | "好的！我非常理解您的需求！让我来为您精心修改这个文件！" |

**红旗词/禁语**：

| 类型 | 禁用 |
|------|------|
| 谄媚 | "太棒了"、"非常好"、"完全正确"、"当然"、"绝对" |
| 冗余 | "让我来帮你"、"我来为您"、"首先我需要" |
| 不确定 | "应该"、"可能"、"看起来"、"大概"、"probably"、"should"、"looks like" |

**执行关联**：hany-implement Step 8（红旗词检测）。

## 规则 8：保持方案简单直接 [优先级：中]

KISS，避免过度工程化。

| | 示例 |
|---|------|
| **正确** | 直接用框架提供的 `useState`，不用自建状态管理 |
| **错误** | 引入 Redux + middleware + saga 来管理一个表单的 3 个字段 |

**触发条件**：方案设计、引入新依赖、创建抽象层时。

**执行关联**：hany-implement Step 2（Constitutional 门禁：简单性 + 反抽象）。

## 规则 9：用户指令优先 [优先级：高]

用户要求与规则冲突时以用户为准。

| | 示例 |
|---|------|
| **正确** | 用户说"用 Write 重写整个文件"→ 执行，虽然规则 4 说优先 Edit |
| **错误** | 用户说"用 Write 重写"→ 拒绝并说"按规则我应该用 Edit" |

**边界**：用户指令明显会导致安全漏洞或数据丢失时，应先警告再执行。

**执行关联**：所有 skill 的用户交互层。

## 规则 10：从重复错误中学习 [优先级：中]

本次对话中用户反复强调的错误必须立即内化避免。

**会话内**：同一错误出现第 2 次 → 必须记录并在后续步骤中避免。

**跨会话**：由各 skill 的"错误→规则抽象"机制处理，写入 `docs/hany/rules.md`。

**规则格式**（与 hany-implement Step 9.1 对齐）：

```markdown
## [日期] 规则 N：[简短标题]
**触发条件**：[什么情况下会犯这个错]
**正确做法**：[应该怎么做]
**反例**：[错误的写法]
**来源**：[哪个 bug/错误抽象出来的]
```

**三级错误分类**：

| 错误类型 | 处理方式 | 持久化位置 |
|---------|---------|-----------|
| 编码习惯问题（如忘处理 null） | 写成通用规则 | `docs/hany/rules.md` |
| 项目特定问题（如某 API 特殊行为） | 写成项目笔记 | 项目文档 |
| 一次性失误（如打错字） | 加测试覆盖 | 测试代码 |

**执行关联**：hany-implement Step 9（过程检查 + 错误→规则抽象）。

## 规则 11：主动修复与优化 [优先级：中]

发现问题直接修，不问"要不要修"。

| | 示例 |
|---|------|
| **正确** | 发现 `getUser()` 没有处理 404 → 直接加上错误处理 |
| **错误** | 发现 `getUser()` 没有处理 404 → 问用户"要不要加个错误处理？" |

**边界**：
- 小问题（null 检查、命名修正、格式问题）→ 直接修
- 可能影响行为的改动（逻辑变更、接口修改）→ 先告知再修
- 超出任务范围的大改动 → 不修，记录到 TODO

**执行关联**：hany-implement Step 7（并行实现中发现周边问题）、所有 skill 的代码审查环节。
