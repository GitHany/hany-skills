# hany-verify-small — 小改动验证

**触发方式：`/hany:verify-small`**

> **调用方**：可独立触发 `/hany:verify-small`，也可由 `hany-auto` Step 4 自动调用。

> 适用场景：小 bug 修复、单个文件修改、紧急热修复。
> 不适用：大功能改动、架构调整、多文件修改（请用 `/hany:verify-project`）

> **核心原则：没有证据就不说完成。运行命令看到输出才能下结论。**

## 质量模式

> 详见 `hany-common/procedures/quality_modes.md`

## 强制交互规则

> 禁止跳过验证直接说"测试通过"，禁止不运行就声称代码能工作。

## 编码约束

> 读取 `hany-common/rules/rules.md`，11 条规则全程生效。

## 工作流程

```
触发 → 先想想看 → 检查改动 → 功能验证 → 修复确认 → 输出报告
```

---

## Step 0：先想想看 [进度 10%]

> ⚠️ **若由 `hany-auto` 调用，跳过此步**（auto 已完成初步思考）。

**触发后，先思考再执行。**

1. 这次修复了什么问题？
2. 改动了哪些文件？
3. 验证重点在哪里？

> 思考完后输出思路再进入 Step 1。

## Step 1：检查改动 [进度 20%]

1. 读取 `git diff`（或用 `git diff --name-only`）确认改动的文件
2. 读取改动内容，确认不是空壳修改
3. 检查语法：运行 `tsc --noEmit` / `python -m py_compile` / `go vet` 等

> **铁律**：Agent 报告"完成" ≠ 真的完成。

## Step 2：功能验证 [进度 50%]

**实际运行验证，不能只看代码。**

| 检查项 | 命令 |
|--------|------|
| 语法/编译 | `tsc --noEmit` / `python -m py_compile` / `go vet` |
| 构建 | `npm run build` / `go build` |
| 测试 | `npm test` / `pytest` / `go test` |
| 功能走查 | 正常输入→预期输出，关键路径通顺 |

**边找边报**：发现阻塞问题立即汇报，其余攒批汇报。

## Step 3：修复确认 [进度 75%]

**确认 bug 真的被修复了。**

1. 写一个能捕获该 bug 的测试用例
2. bug 存在时 → 测试必须失败
3. 修复后 → 测试必须通过

如果无法写测试，用"手动验证步骤"替代。

## Step 4：输出验证报告 [进度 90%]

```markdown
## 小改动验证报告

### 改动文件
- [文件路径]

### 验证结果
| 检查项 | 结果 | 证据 |
|--------|------|------|
| 语法检查 | ✅/❌ | [命令输出] |
| 构建 | ✅/❌ | [命令输出] |
| 测试 | ✅/❌ | [命令输出] |
| 功能验证 | ✅/❌ | [描述] |

### 修复确认
- Bug 是否修复：✅/❌

### 结论
[通过 / 不通过]
```

## Step 5：最终确认 [进度 100%]

用 `AskUserQuestion`：

- **通过** — 任务完成
- **不通过** — 告知失败原因，建议用 `/hany:verify-project` 进一步检查
