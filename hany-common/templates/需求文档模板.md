# 需求文档模板

## 概述

本文档定义需求文档的标准格式，所有通过 `/hany:require` 或 `/hany:auto` 产出的需求文档均使用此格式。

## 目标

1. 所有技能执行后必须生成需求文档
2. 需求文档格式统一、结构完整
3. 文档存储至 `{项目根目录}/docs/hany/requirements/` 目录

## 输入

| 来源 | 描述 |
|------|------|
| 用户原始描述 | 用户通过 `/hany:auto` 或 `/hany:require` 触发的任务描述 |
| 项目上下文 | 项目结构、技术栈、已有代码 |
| 对话澄清 | 多轮提问后的确认信息 |

## 需求文档标准格式

```markdown
---
status: in_progress
author: [作者]
version: "[版本]"
created: [日期]
---

# [主题名称]

## 目标
[一句话描述最终想要什么]

## 详细需求
### 需求项 1
[描述]

### 需求项 2
[描述]

## 输入/输出
| 类型 | 描述 |
|------|------|
| 输入 | [描述] |
| 输出 | [描述] |

## 技术约束
- [约束1]
- [约束2]

## 业务规则
- [规则1]
- [规则2]

## 异常处理
| 场景 | 处理方式 |
|------|---------| 
| [场景] | [处理] |

## 不在范围内
- [排除的功能]

## 待澄清项
- [NEEDS CLARIFICATION: 具体问题]

## 歧义评分
| 维度 | 评分 |
|------|------|
| 目标清晰度 | X% |
| 约束清晰度 | X% |
| 验收标准 | X% |
| 综合 | X% |

## 选中方案
[方案名称 + 理由]
```

## 存储路径

- 路径：`{项目根目录}/docs/hany/requirements/YYYY-MM-DD-<主题>.md`
- 命名规范：日期-简短主题描述
- 示例：`docs/hany/requirements/2026-04-12-用户认证需求.md`
- 模板文件位置：`hany-common/templates/需求文档模板.md`

## 质量标准

| 检查项 | 要求 |
|--------|------|
| status 标记 | 完成后改为 `completed` |
| 无占位符 | TBD/TODO/待定 必须消除 |
| 歧义度 | 综合评分 < 30% |
| 可验证 | 每项需求有明确的验收标准 |

## 触发时机

根据 `hany-require/SKILL.md` Step 9：
- 需求澄清完成后**自动写入** `{项目根目录}/docs/hany/requirements/`
- 无需确认，直接写入后输出摘要

## Skill 协同规范

### hany-auto 与 hany-require 的需求文档协同

| 阶段 | Skill | 产出 |
|------|-------|------|
| Step 2 | hany-require | 需求文档 `docs/hany/requirements/` |
| Step 2.4 | hany-auto | 摘要压缩，引用需求文档路径 |
| Step 4.4 | hany-auto | 归档汇总，引用需求文档路径 |

### 拆分规则（hany-require Step 4）

需求涉及多个子系统时：
- 每个子系统生成独立需求文档：`docs/hany/requirements/YYYY-MM-DD-<主题>-<子系统>.md`
- 命名格式：`2026-04-12-用户模块-认证.md`、`2026-04-12-用户模块-权限.md`

### 摘要格式（hany-auto Step 2.4）

```markdown
[require 摘要] 需求：[一句话描述]。需求文档：docs/hany/requirements/XXX.md。选中方案：[方案名]。歧义评分：[最终综合分]。待澄清：[剩余标记数]。
```

### 快速路径（hany-auto 简化流程）

简单任务走快速路径时：
- 需求文档可简化为 YAML frontmatter + 一句话目标 + 核心约束
- 但必须保留歧义评分和选中方案

## 各 Skill 产出要求

### hany-require 需求文档产出（Step 9）

| 检查项 | 要求 |
|--------|------|
| 路径 | `docs/hany/requirements/YYYY-MM-DD-<主题>.md` |
| status | 初始 `in_progress`，完成后 `completed` |
| 歧义评分 | 必须 < 30% 才能写入 |
| [NEEDS CLARIFICATION] | 写入前必须全部消除 |
| 示例原型 | Step 10 产出作为附录附在文档末尾 |

### hany-auto 需求文档引用（Step 2.4 & 4.4）

| 检查项 | 要求 |
|--------|------|
| Step 2.4 | 摘要中必须引用需求文档路径 |
| Step 4.4 | 归档中必须列出需求文档路径 |
| 恢复检查 | Step 0 前检查是否存在 `in_progress` 状态文档 |

### 快速路径需求文档简化版

```markdown
---
status: in_progress
author: [作者]
version: "1.0"
created: [日期]
---

# [主题名称]

## 目标
[一句话]

## 约束
- [约束1]

## 选中方案
[方案名 + 理由]

## 歧义评分
| 维度 | 评分 |
|------|------|
| 综合 | X% |
```

## 归档

完成后将 `status: in_progress` 更新为 `status: completed`。
