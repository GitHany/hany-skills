# 拆分并行处理规则（公共）

需求拆分为多个子系统后，各阶段统一遵循：

1. 每个子系统独立进入对应阶段流程
2. 无依赖的子系统用 Agent/Team **并行执行**
3. 有依赖的子系统按依赖顺序执行（被依赖方先完成）
4. 统一进度汇报：`[阶段进度] 子系统A：X% / 子系统B：Y% / 总体：Z%`

## 并行决策检查（强制）

> **并行不是银弹，依赖关系决定能否并行。**

启动并行组之前，必须分析文件依赖关系：

| 依赖强度 | 判定标准 | 并行策略 |
|---------|---------|---------|
| **强依赖** | A文件被B文件直接引用（如import、include、继承）；修改A会导致B必须同步修改 | **不并行**，串行执行 |
| **中等依赖** | A和B都依赖同一个模块，但彼此不直接依赖 | 可并行，但需统一接口 |
| **无依赖** | 两个文件完全独立，修改互不影响 | **可以并行** |

**并行决策流程：**
```
启动并行组前：
1. 列出每个并行组会修改的文件清单
2. 检查文件间是否存在 import/include/继承/引用 关系
3. 判断依赖强度
4. 如果强依赖 → 不并行，告知用户原因："[文件A]和[文件B]存在强依赖（[原因]），串行执行更安全"
5. 如果无/中等依赖 → 可以并行
```

## 非代码任务的并行判断

| 任务类型 | 并行可行性 | 说明 |
|---------|-----------|------|
| 纯文档编写（.md） | ✅ 可并行 | 无运行时依赖 |
| 配置文件修改 | ⚠️ 看情况 | 如果改同一个文件则不行 |
| 测试用例编写 | ✅ 可并行 | 独立文件，通常无依赖 |
| 样式/CSS修改 | ✅ 可并行 | 独立文件 |
| API接口修改 | ❌ 需串行 | 接口定义和实现通常强耦合 |

> **告知用户原则**：如果决定不并行，必须说明原因，让用户理解是为了保证质量。
